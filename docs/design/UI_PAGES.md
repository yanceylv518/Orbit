# Orbit 控制台页面结构重构设计

> 目标：让页面组织表达两件事——**第一阶段的主工作流**（同步 → 生成计划 → 审查 → 确认/导出）与**策略核心认知**（Δ 净敞口、生命周期相位、锚点与触发距离）。
> 替换现有 Vue 前端对旧控制台的 1:1 复刻结构。上游：`ARCHITECTURE.md` §7（目标态 IA）、`STRATEGY_LOGIC.md`（概念定义）。
> 前端技术栈维持 Vue 3 + Vite（`frontend/src/`），本文只重排页面与信息层级，不换栈。

最后更新：2026-07-12

---

## 0. 设计原则

1. **页面组织 = 管理员的工作流**，不是数据表的镜像。第一阶段主线"同步 → 生成 → 审查 → 确认/导出"必须在工作台一屏可见、逐步可点。
2. **核心概念上墙**：Δ、相位、锚点、偏离与触发距离——后端 `net_exposure_v1` 内核已产出这些字段，页面必须消费它们，否则认知只存在于文档。
3. **模式感知**：`read_only` 模式下不出现任何模拟盘控件（Tick/暂停）；顶栏主动作随模式切换。
4. **每个数字可解释**：任何 KPI、徽章、拦截点进去都能看到来源（计划 → 触发快照，风险 → 审计记录）。
5. **空态给动作**：每张空表都指出下一步（"先同步账户"→ 按钮直达）。

## 2. 全局框架

```
┌ 侧栏 ────────┐ ┌ 顶栏 ──────────────────────────────────────────────┐
│ 7 菜单        │ │ 页面标题 · [模式徽章 READ_ONLY] · [风控状态徽章→风控中心] │
│              │ │ 主动作（随模式）:                                    │
│ 操作者卡      │ │   read_only: [同步全部账户] [生成执行计划]            │
│ 用户/模式/退出 │ │   mock(dev): [执行 Tick] [暂停]（仅 mock_data_enabled）│
└──────────────┘ └────────────────────────────────────────────────────┘
```

- 模拟盘控件（Tick/暂停）只在 `mock_data_enabled=true` 渲染——只读模式下它们是误导性噪音。
- 风控状态徽章常驻顶栏（normal / watch / stopped），点击进风控中心；急停按钮**只**放风控中心（防误触）。

## 3. 页面规格

### 3.1 工作台（重做）

现状问题：KPI 是模拟盘指标（总权益/今日盈亏），只读阶段大多为 0；主流程漏斗不可见。

```
┌ 漏斗 KPI（逐格可点击直达对应页）────────────────────────────────────┐
│ [账户同步 2/3] [Hedge 通过 2/2] [计划待确认 4] [风控拦截 1] [无动作 2] │
├ 待办区 ──────────────────────────┬──────────────────────────────────┤
│ 同步失败账户                      │ 待确认计划（前 5 条，直达执行计划页）│
│  binance_x ❌ 签名错误 [重试同步]  │  plan_… BTCUSDT REDUCE_SHORT [审查] │
├ 系统状态行 ──────────────────────┴──────────────────────────────────┤
│ 模式 read_only · 存储 MySQL · 内核 net_exposure_v1 · 最近同步 12:03:22 │
└─────────────────────────────────────────────────────────────────────┘
```

数据绑定（现有 snapshot 全够）：`exchange_accounts[].status/credential_error` + `binance_account_snapshots[].status/position_mode.hedge_mode_ok/synced_at` + `execution_plans[].status/manual_review`。
权益/盈亏指标降为系统状态行的次要信息（等 paper 模式再升回 KPI）。

### 3.2 用户与账户（微调）

保持两区结构（用户列表 / 账户列表）。补三点：
1. 账户行加"最近同步时间 + 状态徽章"（`synced_at` 已有）；
2. 凭证保存成功后就地给"去同步"按钮（工作流衔接）；
3. Hedge Mode 不通过的账户行整行醒目标记（对应 PROGRESS 下一步·工程 4）。

### 3.3 策略中心（EventsPage 升级更名）

| 区块 | 内容 | 数据 |
|---|---|---|
| 策略实例卡 | 名称/版本/模式/状态/覆盖 symbols —— 强调"平台策略，系统维护" | `strategy` |
| 账户挂载表 | account × {enabled, mode, symbols, budgets, max_single_order, allow_*, updated_at} + 行内编辑 | `account_run_configs`（API 已有，UI 缺失挂载视图） |
| 事件参数区 | 现三卡保留；**未接线参数标注"未生效"**（STRATEGY_LOGIC §8 映射）；σ 化后此区随内核换单位并展示 C1–C9 校验结果 | `event_config` |
| 内核状态行 | `exposure_model` 版本、生命周期/RiskGuard 已启用能力清单 | plans trigger / 常量 |

### 3.4 执行计划（增强）

保留现有漏斗摘要 + 确认/导出。补：

1. **详情展开行**（PROGRESS 下一步·工程 5）：点行展开——
   - 触发快照：`trigger.exposure_model / 当前净敞口 / 目标净敞口 / delta_to_target_qty`、mark_price、双腿数量与 PnL、预算；
   - 生命周期上下文：`trigger.event_rule / 生命周期状态 / 趋势退出计数`（后端已写入）；
   - 动作明细与风控逐条结果。
2. 列增加：**相位**（symbol state）与 **Δ→Δ\***（一列显示 `当前 → 目标`）。
3. 拦截计划的 `block_reason` 用徽章前置，不埋在动作文本里。

### 3.5 币种视图（升级——策略大脑可视化）

现状问题：只有价格/多空/浮盈，Δ、相位、锚点、触发距离全部缺席，而**这些数据后端已经有**。

每币种卡片头（替代现顶部指标条）：

```
┌ BTCUSDT ── [相位: TREND_UP] ─────────────────────────────────┐
│ 价格 60,900 · 锚点 60,000 · 偏离 +1.50%                       │
│ 触发进度: 0 ────●──── a_pt(1.5%) ───────────── θ_t(4%)        │
│ Δ 净敞口 −0.004 (≈−243 USDT) · Δ* 目标 −0.004 · 差 0          │
│ 多 0.016 / 空 0.020 · 本轮计数: 搬运1 减仓0 恢复0              │
│ 趋势极值 61,200 · 退出确认 2/3 tick（TREND 相位时显示）        │
└──────────────────────────────────────────────────────────────┘
```

数据绑定：`symbol_states` 的 `state / base_price / base_qty / trend_extreme_price / trend_exit_candidate_count / *_count_in_trend`；Δ = `long_qty − short_qty`（前端可算）；触发线取 `event_config`（现为固定 %，σ 化后换算注明）。
下方保留：仓位概览表（加 Δ 列、按账户/币种筛选——PROGRESS 下一步·工程 4）、图表、事件时间线。

### 3.6 风控中心（增强）

| 区块 | 第一批（数据已有） | 第二批（等内核补齐） |
|---|---|---|
| 风控 KPI | STOPPED / ONLY_REDUCE 状态的 symbol 数、拦截计划数 | 组合级回撤、快照新鲜度 |
| 拦截分类 | 三桶：账户同步风险 / Hedge 风险 / 计划动作风险（PROGRESS 下一步·工程 6） | — |
| 预算与账本 | — | C5/C6 回撤预算消耗条、C7 自融资账本余量 |
| 准入 | — | 各 symbol π̂ 准入表（Phase B 产出） |
| 审计 + 急停/恢复 | 保留；STOPPED 的 symbol 提供"人工复核恢复"入口占位 | 复核流程完善 |

### 3.7 报表（合并降权）

Tab 结构：`复盘日报 | 事件日志`。只读阶段两者数据稀少，空态说明"依赖 paper 模式产生完整数据"。LogsPage 组件并入，菜单项删除。

## 5. 交付批次

**第一批（数据全部已有，纯前端）**：菜单重组 7 项、工作台漏斗、顶栏模式感知、币种卡片头（相位/Δ/锚点/触发进度）、计划详情展开行 + 相位/Δ 列、拦截三桶、报表合并、未生效参数标注。

**第二批（跟随内核缺口补齐，见 PROGRESS「策略逻辑设计落地」）**：C5/C6 预算消耗条、C7 账本、组合级回撤与快照新鲜度 KPI、π̂ 准入表、σ 化阈值展示与 C1–C9 校验结果。

验收基准（第一批）：管理员打开工作台，30 秒内能回答——哪些账户没同步上、为什么；有几个计划等我确认；有没有 symbol 被风控停了。打开币种视图，能直接读出每个 symbol 处于三环闭环的哪个环节、离下一个触发还有多远。
